# OP041 — Testing Actions with Harness

| Field | Value |

Added in [PR #1005](https://github.com/canonical/operator/pull/1053)

## Abstract

The ops.testing.Harness testing framework currently has limited support for testing the action functionality of charms. We expect Harness to provide support for thorough unit testing of charms, which includes actions. This spec proposes adding a `Harness.run_action` method to allow testing charm actions.

## Rationale

The [ops.testing.Harness](https://ops.readthedocs.io/en/latest/#ops.testing.Harness) supports a low-level way to emit action events, but testing requires patching. A minimal test could look like:

```py
@patch.dict(os.environ, {"JUJU_ACTION_NAME": "my_action"})
def test_my_action(harness):
    with patch.object(harness.charm.framework.model._backend, "action_get"):
        harness.charm.on.my_action.emit()
```

Any parameters for the action cannot easily be provided, and the charmer does not have an easy mechanism to get access to any results set by the charm[^1]. In addition, the `JUJU_ACTION_NAME` environment variable must be set[^2], and some implementation for `action_get()` must be provided.

We expect Charmers to have reasonable unit test coverage of their charms, and actions are a common part of charms, so we should provide a more convenient mechanism for unit testing charm actions.

## Specification

A new method will be added to `ops.testing.Harness`, `run_action`, that allows simulation of running an action (e.g. with `juju run`). In addition, a new data class will be added to `ops.testing`, `ActionOutput`, which provides the structure for the logs, and result of the action; and a new Exception class, `ActionFailed`, which is raised when `fail()` is used.

```py
@dataclasses.dataclass(frozen=True)
class ActionOutput:
    logs: list[str]
    """Messages generated by the Charm using :meth:`ActionEvent.log`."""

    results: dict[str, str]
    """The action's results, as set or updated by :meth:`ActionEvent.set_results`.

    The results dictionary is flattened to a dict[str, str] when communicating with
    Juju, and Juju restores it back to a multi-level dict. This attribute has gone
    through the same flatten-unflatten process, so may be a multi-level dict, and
    calling :meth:`ActionEvent.set_results` may raise :class:`ValueError` if the
    results cannot be flattened.

    Multiple calls to :meth:`ActionEvent.set_results` update the dictionary rather
    than replacing it.
    """

class ActionFailed(Exception):
    message: str
    """Optional details of the failure, as provided by :meth:`ActionEvent.fail`."""

    output: ActionOutput
    """Any logs and results set by the Charm."""

class Harness(Generic[CharmType]):
    ...
    def run_action(
        self,
        action_name: str,
        params: dict[str, Any]|None = None,
    ) -> ActionOutput:
        """Simulates a `juju run` command.

        Args:
            action_name: the name of the action to run, as found in ``actions.yaml``.
            params: override the default parameter values found in ``actions.yaml``.
                If a parameter is not in ``params``, or ``params`` is ``None``, then
                the default value from ``actions.yaml`` will be used.

        Raises:
            ActionFailed: if :meth:`ActionEvent.fail` is called. Note that this will
              be raised at the end of the ``run_action`` call, not immediately when
              :code:`fail()` is called, to match the run-time behaviour.
        """
        ...
```

This method ensures that the `JUJU_ACTION_NAME` environment variable is appropriately set, and (unless hooks are disabled) emits the appropriate action event. The result of the action and any values passed via `ActionEvent.log()`, if any, are returned from the method. If the Charm calls `ActionEvent.fail()` then this is tracked and an `ActionFailed()` exception raised at the end of `run_action` (this mimics the run-time behaviour where the Charm code continues after `fail()`).

When testing handling of a charm's configuration, a Charmer uses `harness.update_config()` and provides only the configuration values that have changed (if any) from the default (or previous) values. Action parameters have less state - parameters do not persist from run to run - but have similar default values. For `run_action`, if a parameter is not provided, either because no params dictionary is provided or it does not have the relevant key, then the default value from the metadata is used.

Simple unit tests for an action could look like:

```py
def test_action(harness):
    out = harness.run_action("my-action", {"arg": "value"})
    assert out.results == {"key1", "value1", "main": {"sub1": "sval1", "sub2": "sval2"}}
    # We expect these would not typically form part of the defined interface of the
    # charm, so would be less likely to be tested, but are available if needed.
    assert out.logs == ["message 1", "message 2"]

def test_action_failed(harness):
    with pytest.raises(ops.testing.ActionFailed) as e:
        harness.run_action("my-failing-action")
    assert e.message == "explanation"
    # We expect that when testing failure, the fact that it failed is likely
    # the only concern, and possibly also the failure message. However, if
    # `set_results()` or `log()` had been called in addition to `fail()`, then
    # those are also available.
    assert e.output.logs == ["Reached step 1"]
    assert e.output.results == {"partial": "value"}
```

When running an action, the action is run on a specific unit; for example, `juju run testing-actions/0 my-action`. However, `Harness` generally models a single unit (e.g. `exec()` and `set_leader()` don't take a unit name). For consistency, `run_action()` will behave in the same way, and if an action would execute differently depending on the unit, the Charmer should configure the `Harness` object appropriately before calling `run_action()`.

## Further Information

### Additional Notes

This functionality was requested by a community member in [canonical/operator:#762](https://github.com/canonical/operator/issues/762).

#### Comparative Implementation

The [Scenario testing framework](https://github.com/canonical/ops-scenario) includes [support for testing actions](https://github.com/canonical/ops-scenario#actions), for example:

```py
from scenario import Action, Context, State, ActionOutput
from charm import MyCharm

def test_backup_action():
    # Define an action.
    action = Action("do_backup", params={"a": "b"})
    ctx = Context(MyCharm)

    # If the parameters (or their type) don't match what declared in actions.yaml,
    # the `ConsistencyChecker` will slap you on the wrist.
    out: ActionOutput = ctx.run_action(action, State())

 # You can assert action results, logs, and failure.
    assert out.results == {"foo": "bar"}
    assert out.logs == ["baz", "qux"]
    assert out.failure == "boo-hoo"

    # ...
```

### Alternatives

#### How to indicate the success or failure of the action

Several other possibilities were considered for letting the Charmer know whether the charm called `ActionEvent.fail()`, and if so what the optional failure message was. These included:

```py
out = harness.run_action("my-action")
assert out.status == "completed"  # Too error prone.
assert out.status == ops.testing.ActionStatus.COMPLETED  # Too wordy.
assert out.failure is not None  # Matches Scenario, slightly long, easily confused with ↓
assert not out.failure  # The message is optional, so would need to be elsewhere.
assert out.success  # Short, but easily forgotten.
```

An action can have several states (*running*, *completed*, *failed*, *error*), but when writing unit tests we expect Charmers are only concerned with *completed* and *failed*. Our expectation is that the most common check in an action unit test will be that it succeeded, so we want to optimise that code path. We also want to avoid simple errors, such as failing to test that the action succeeded.

#### Re-evaluate after higher-level testing discussion completes

An alternative choice would be to wait until the current discussions around how Charmers should approach charm testing (e.g. Harness vs. Scenario, updating pylibjuju and pytest-operator) have completed - if the recommendation going forward will be to use (some form of) Scenario, then it may not be worth solving this need in Harness. If that approach is chosen, it would be worthwhile making this clearer to Charmers; for example, by bypassing the `JUJU_ACTION_NAME` check and providing a suggestion to use Scenario in the `NotImplementedError` raise.

#### Detailed Example Code

A charm that exercises the test examples outlined in this specification is available in the author's [testing-actions-operator repository](https://github.com/tonyandrewmeyer/testing-actions-operator). A set of tests using ops.testing.Harness and unittest, using ops.testing.Harness and pytest, using ops-scenario, and using pytest_operator is provided to demonstrate the current state of charm action testing. A [branch](https://github.com/tonyandrewmeyer/testing-actions-operator/tree/with-proposed-harness-run-action) and [PR](https://github.com/tonyandrewmeyer/testing-actions-operator/pull/1) that convert the tests to use the proposed API are included.
