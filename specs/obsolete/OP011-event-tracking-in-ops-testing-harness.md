# OP011 — Event tracking in ops.testing.Harness

| Field | Value |
| --- | --- |
| Status | Rejected |
| Type | Standard |

## Problem statement

Users want a way to look at the events fired during a test and examine them.  A concrete case is to see if events were deferred by callbacks or not.

Original issue: [https://github.com/canonical/operator/issues/433](https://github.com/canonical/operator/issues/433)

## Background

The original issue proposes two things:

* Have a way to determine while testing whether an event was deferred.
* Have a generic way of examining events fired during tests.

However, events can be handled by multiple observer callbacks.  And events can be deferred on a per-callback basis.

Thus, we really could be talking about 2 things: the events themselves, and/or the individual "notices", which are what OF calls it when an event triggers an observer callback.

[note: code uses "monitor" and "observer" terms; probably want to review and update appropriately, or even note details re: the terminology.]

As per-notice callbacks can be deferred, OF stores events and notices in a database, dropping them when they're fully handled, but otherwise keeping them around to re-fire on subsequent hooks.

Possible solutions need to not just consider the events which are fired, but the fact that those specific Python objects may disappear after they're serialized to the database, and thus it may be more appropriate to operate off the objects in the database.

## Currently preferred solution

Presently I would suggest we provide a specialized storage class which is updated to, instead of dropping events/notices on completion, instead flags those events as being fully handled or not.  APIs used by the non-test code would still be expected to return values as if fully-handled events/notices were dropped from the database, however APIs would either be extended or extra non-public functions added to allow ops.testing.Harness to pull the full sets of events/notices for examination by the test author.

Regarding APIs, I think we should at least expose a list of notices, linked to the events, with per-notice indication of whether the notice has been handled or deferred.  We could also add optional filtering of notices by event type, which would be a simple list comprehension or generator expression filter based on the type of event associated with the notices.

We could also have a separate API which exposes just the events, and omits the details of the notices, instead just giving an event-level handled-or-deferred indication.  This could be useful for introspecting the data associated with custom events, for example.

In summary and somewhat more concretely, we could do the following:

* Create a new class, ops.testing._TestingStorage, which inherits from ops.storage.SQLiteStorage, adds "deleted" columns to both the "snapshot" and "notice" tables of the base class, and updates the read method(s) of this table to only return records which haven't been marked as deleted.  In other words, it should function like the normal SQLiteStorage class except that true deletion is replaced by a deleted flag.
* Add the following methods to the Harness class:
  * get_event_notices(self, **filters): returns a list, from oldest to newest, of all the event notices generated by events.  Each item in the list would include:
    * The event_path, observer_path, and method_name data from the notices table.
    * A reference to the event object, as loaded from its DB snapshot.
    * A deferred flag, which is True if the record has not been flagged as deleted (i.e. handled), or False otherwise.

    The filters are to be determined, but it could be things like the original BoundEvent class, references to a specific callback function object itself, and/or the event_path/observer_path/method_name strings.

  * get_events(self, **filters): returns a list of the snapshotted event objects, as loaded from the database, with the EventBase.deferred attribute appropriately set to reflect whether the event has been already handled or not.
